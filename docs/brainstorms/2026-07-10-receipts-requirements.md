# Receipts — a runtime PR-verification agent

## Summary

Receipts is a standalone tool that verifies a code change by **running the application**, not by reading the diff. On a change, it builds and runs the target app from source, has an AI agent drive the affected user flows in a real browser against the real backend, asserts both the UI and the backend effects, and produces a reproducible evidence report — video, screenshots, the offending lines, repro steps, severity, and whether the failure is caused by this change or pre-existing. v1 is a working single-repo CLI tool that ships with a bundled full-stack sample app carrying planted runtime bugs, with clean seams for a GitHub-PR front-end, a Docker runtime, and richer reporting later.

This is a separate project from DocuMind (its own architecture; it does not reuse `documind`), living as a top-level `receipts/` package in this repository.

---

## Problem Frame

Static review tools (CodeRabbit, Cursor bugbot, Greptile) reason over the text of a diff. They cannot see the failure class that only appears when the app runs: a reorder that updates local UI state but never persists to the API, a token refresh that runs after a redirect and drops a returning user into a login loop, a rollback path that crashes mid-migration, an integration whose real response differs from its mock. These read cleanly in the diff, pass static checks, and survive review — then break for a real user. That is not a code-quality problem; it is a **visibility** problem.

Receipts closes that gap by making runtime behavior observable on every change. The win condition is not "an agent clicked around" — it is **reproducible evidence a developer can act on without re-investigating**: the receipts. Everything in v1 serves that: run the real app, drive the changed flow, assert what actually happened, and hand back a bundle anyone can rerun.

## Key Decisions

- **Runtime verification is the whole point.** The tool's value is running the app and observing behavior, so v1 commits to really building and running one target and driving it full-stack (real browser + real backend), not simulating or reasoning statically. Everything else is deferred to protect this core.

- **Standalone architecture, not a DocuMind module.** Receipts has its own package (`receipts/`), its own models, and does not reuse `documind`'s LLM client. It lives in this repo to stay in scope, but shares no code with the learning portfolio.

- **Stack: Claude + Playwright, minimal deps, no LangChain.** Python 3.11, the `anthropic` SDK directly (Claude Opus 4.8 for planning and hard driving reasoning, Claude Sonnet for cheap per-step decisions), Playwright for browser driving and video/screenshots, Pydantic v2 for structured plans/reports, Typer for the CLI. LangChain is deliberately excluded: the driver is a bespoke browser-tool loop where a framework adds indirection without payoff.

- **Pluggable `Runtime`; Docker is faithful, Subprocess is how we demo.** A `Runtime` interface abstracts build/run/teardown. `DockerRuntime` (fresh single-use container per run) is the faithful, primary implementation and is built and unit-tested. `SubprocessRuntime` (host processes) exists because the Docker daemon is not reachable in the development sandbox — it is what runs the live end-to-end demo here. Both honor one interface, so the Docker path is exercised wherever a daemon exists without changing anything downstream.

- **CLI first, GitHub second.** v1 is invoked as `receipts review <path> --base <ref>` against a local git checkout, with the diff coming from git. A `DiffSource` seam lets a GitHub-PR front-end (fetch diff + description, post a comment) wrap the same core later.

- **Explicit `receipts.yaml`, not "zero-config."** Auto-detecting arbitrary build/run/auth is a research project. v1 requires the target to declare build command, run command, health check, auth/seed steps, and backend probes in a `receipts.yaml`. This is honest: "works on any repo that adds one file."

- **Planner is agentic and steerable.** The plan is generated by the LLM from the change plus an app map, and is steerable by plain-English hints (in config or a CLI flag) that add, prioritize, or suppress scenarios — mirroring Ito's real feature.

- **Driver = agentic execution + self-healing selector cache.** Execution is LLM-agent-driven per step (flexible, faithful to how Ito navigates), but every resolved action is cached as `intent → concrete locator + input`. On replay the cache drives execution deterministically **without the LLM**; the LLM re-engages only on a cache miss or a stale/failed selector (self-heal), bounded by a configurable budget (default 3 attempts/step). The cache **is** the deterministic replay artifact, so agentic discovery and reproducible replay come from one mechanism.

- **Layered pass/fail, not LLM-only.** A scenario's verdict prefers explicit machine checks — DOM/text assertions and declared backend probes — and only falls back to an LLM-judge over the final state when those cannot express the expectation. This keeps findings trustworthy and false positives low.

- **Backend effects are asserted via declared probes.** The half a browser cannot see (DB writes, business logic) is checked through HTTP endpoints and/or read-only DB queries declared in `receipts.yaml`.

- **PR-attribution by base-branch replay.** A failing scenario is replayed against the base branch; failing on both = pre-existing, failing only on the change = PR-attributable.

- **Deterministic tests via fakes; agentic path via live demo.** Following the DocuMind convention, unit tests inject a fake LLM and a stub/fixed target so the suite runs offline with no API key and no Docker. The real browser + real Claude path is exercised by a separately-marked live demo and integration test.

## Requirements

**Diff & trigger**

R1. A CLI `receipts review <path> --base <ref>` produces, from a local git checkout, a change set (changed files + unified patch) and a change description (from commit messages, overridable with `--desc`).

R2. Diff acquisition sits behind a `DiffSource` interface so a GitHub-PR source can be added without touching downstream stages.

**Target configuration**

R3. A target declares its build command, run command, readiness/health check, and base URL/port in a `receipts.yaml` at its root.

R4. `receipts.yaml` also declares auth/seed setup (how to obtain an authenticated session and how to seed baseline data) and backend probes (named HTTP endpoints and/or read-only DB queries) usable for assertions.

R5. Configuration is schema-validated with clear, actionable errors on missing or malformed fields.

**Runtime**

R6. A `Runtime` interface builds the target from source, starts it, waits until healthy, exposes a base URL, and tears down — with `DockerRuntime` (fresh container per run) and `SubprocessRuntime` (host processes) implementations honoring the same interface.

R7. The runtime is selectable via config/flag; every run is isolated and torn down even when a build, start, or scenario fails.

**Planner**

R8. Given the change set, the change description, and an app map, the planner produces a structured `TestPlan` of `Scenario`s — each with a name, rationale, target flow, expected outcome, an adversarial flag, and the changed lines that motivate it.

R9. The plan is steerable by plain-English hints (from `receipts.yaml` or a `--hint` flag) that add, prioritize, or suppress scenarios.

R10. The plan focuses on flows the change touches — happy path plus adversarial cases (double-submit, expired session, boundary inputs, permission crossing) — rather than a full regression sweep.

**Driver**

R11. For each scenario, an agentic driver operates a real browser (Playwright), choosing actions from observed page state to accomplish the flow, while recording video, screenshots, and a structured action log.

R12. Every resolved action is cached as `intent → concrete locator + input`. On replay the cache drives execution deterministically with no LLM call; the LLM re-engages only on a cache miss or a stale/failed selector (self-heal), bounded by a configurable per-step budget (default 3).

R13. Backend effects are asserted through the declared probes (HTTP/DB) at the point in the flow where the change is expected to have taken effect.

**Verdict & evidence**

R14. A scenario's verdict is layered: explicit DOM/text assertions and backend probes decide pass/fail when they can express the expectation; an LLM-judge over final state is the fallback only when they cannot.

R15. Each scenario emits a deterministic replay artifact (the resolved selector cache plus inputs) that reruns without the LLM and reproduces the same actions.

R16. A finding is labeled PR-attributable vs. pre-existing by replaying the failing scenario against the base branch and comparing outcomes.

**Report**

R17. A run produces a markdown report: per-scenario pass/fail, a severity rating, the changed lines responsible, plain-English repro steps, and links to the evidence bundle (video, screenshots, action log).

R18. Reporting sits behind a `Reporter` interface (markdown to stdout + file in v1) so a GitHub-PR-comment reporter can be added later.

**Re-verify**

R19. Given a prior run and a new commit, re-verify re-runs only the scenarios whose changed lines/flows are affected and reports whether each prior failure is resolved.

**Conventions & deliverables**

R20. A bundled full-stack sample app (FastAPI backend + a small frontend, with an auth flow) ships with its own `receipts.yaml` and 2–3 planted runtime bugs of the class the tool targets, used by the demo.

R21. Deterministic unit tests inject a fake LLM and a stub/fixed target so the whole suite runs offline with no API key and no Docker; the agentic browser path is exercised only by a separately-marked live demo/integration test.

R22. `receipts review ./sample_app --base <ref>` runs the full pipeline end-to-end against the sample app via `SubprocessRuntime` and writes the markdown report; a `--demo` narrates the caught bugs.

R23. A project README plus a short write-up capture the architecture, the seven seams, and what was non-obvious.

## Key Flows

F1. **Full review pipeline.**
**Trigger:** `receipts review <path> --base <ref>`.
**Steps:** `DiffSource` yields the change set + description → planner builds the `TestPlan` (steered by hints) → `Runtime` builds and starts the target and waits for health → for each scenario the driver drives the browser, self-healing via the selector cache, and runs backend probes → the layered judge sets each verdict → base-branch replay attributes failures → the `Reporter` emits markdown + evidence → runtime tears down. **Covers R1, R6, R8, R11, R13, R14, R16, R17.**

F2. **Agentic drive with self-healing cache (within a scenario).**
**Trigger:** a scenario begins.
**Steps:** for each step, if a cached `intent → locator+input` exists, execute it deterministically; on a hit that fails or a miss, ask the LLM to resolve the action from observed page state, execute it, and update the cache; stop the step after the self-heal budget is exhausted (scenario fails with evidence). **Covers R11, R12, R15.**

F3. **PR-attribution via base-branch replay.**
**Trigger:** a scenario fails on the change.
**Steps:** rebuild/run the base branch, replay the same scenario from its cache → fails on base too ⇒ pre-existing; passes on base ⇒ PR-attributable. **Covers R16.**

F4. **Re-verify on a pushed fix.**
**Trigger:** `receipts review` on a new commit with a prior run present.
**Steps:** diff the new commit against the prior run, select only affected scenarios, replay them, and report which prior failures are now resolved. **Covers R19.**

## Acceptance Examples

AE1. **Covers R11, R13, R14, R17.** On the planted "reorder never persists" bug, Receipts drags to reorder (UI shows the new order), the declared `GET /api/items` probe still returns the old order, and the scenario **fails** with video, the offending lines, and repro steps.

AE2. **Covers R10, R11.** On the planted "token refresh after redirect" bug, the returning-user-with-stale-session scenario lands in a login loop and **fails**.

AE3. **Covers R8, R17.** A benign change (copy-only) yields scenarios that all **pass**; the report is green with the flows exercised.

AE4. **Covers R12, R15.** Re-running a passed scenario drives entirely from the cached selectors with **zero LLM calls** and reproduces the same result.

AE5. **Covers R16.** A scenario that fails on the change and also fails when replayed on the base branch is labeled **pre-existing**, not PR-attributable.

AE6. **Covers R21.** The full unit-test suite passes with a fake LLM and a stub target — **no network, no API key, no Docker**.

AE7. **Covers R22.** `receipts review ./sample_app --base main` runs end-to-end via `SubprocessRuntime` and writes a markdown report that catches the planted bugs.

## Scope Boundaries

Deferred to later versions (built as seams only, not implemented in v1):

- GitHub-PR front-end and PR-comment posting (`GitHubPRDiffSource`, GitHub `Reporter`).
- Zero-config auto-detection of build/run/auth (replaces the explicit `receipts.yaml`).
- Browser swarm / scenario parallelism; multi-tenant; a hosted dashboard.
- Remote / Kubernetes runners; a hosted build farm.
- Learned per-organization testing preferences.

Outside v1 entirely:

- Live end-to-end exercise of `DockerRuntime` in this sandbox — the daemon is unreachable here; `DockerRuntime` is built and unit-tested, and its live run happens wherever a daemon exists.
- Non-web targets (mobile, desktop, pure CLI apps).
- Auth flows Receipts cannot script from `receipts.yaml` (e.g., real third-party OAuth with MFA).
- A production-grade severity model; v1 uses a simple, documented rubric.

## Dependencies / Assumptions

- **Runtime env:** Python 3.11; the `anthropic` SDK; Playwright (Python) with the preinstalled Chromium and ffmpeg (video needs ffmpeg — present); Pydantic v2; Typer; PyYAML; Docker CLI/SDK for `DockerRuntime`. Node 22 for the sample app's frontend build.
- **Credits:** the planner, driver, and LLM-judge require Anthropic credits; Sonnet handles cheap per-step decisions, Opus the planning and harder reasoning.
- **Docker daemon is not reachable in the development sandbox** — the live end-to-end demo runs via `SubprocessRuntime`; the Docker path is validated where a daemon exists.
- **Determinism assumption:** the sample app and stub target are deterministic enough for the offline suite; the agentic browser path is inherently non-deterministic and is covered by the live demo, not the offline suite.

## Outstanding Questions

Deferred to planning:

- Selector-cache key scheme (scenario+step index vs. semantic intent hash) and the exact self-heal budget default (proposed: 3 per step).
- How the app map is produced (static route/component scan vs. an LLM summary of the repo) and how much of it feeds the planner.
- Backend DB-probe access mechanism (a read-only connection string in `receipts.yaml` vs. exec-a-query inside the container).
- The severity rubric (dimensions and thresholds).
- Sample-app frontend choice: React/Vite (realistic, heavier Node build) vs. a minimal vanilla-JS SPA (lighter, still full-stack) — pick the one that keeps the demo fast without weakening the browser story.
- Model-routing thresholds (when Opus vs. Sonnet in the driver loop).

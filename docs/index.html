The gate — a verdict on every pull request

Deterministic and reason-backed. The same signal a junior engineer and the board can both read.

Green — merge automaticallyAll journeys pass, the tests grade clean, and the selector cache is unchanged or the change is justified by the requirement.
Amber — request reviewPassing, but something wants human eyes: an unjustified selector change, a middling trust score, or a coverage gap.
Red — raise a bugA journey failed or a selector was hallucinated. klew files a ticket with reproduction steps and links it to the story.

The complete list

Everything klew supports

The full catalogue, grouped by what it does. Every capability below is built, tested and in the repository today.

01 Exploration & selectors

  • Token-efficient driving of a live UI via the Microsoft Playwright CLI.
  • User-facing selector policy — role and label first, automation-id next, CSS only as a last resort.
  • Active-tab root scoping so selectors never collide across open tabs.
  • Confidence scoring and accessibility flags on every resolved selector.
  • Snapshot diffing to keep exploration lean on tokens.

02 Governance & cache

  • Human-approved selector cache — nothing is stored without sign-off.
  • Delta-only updates, so a pull request shows just what genuinely changed.
  • Dry-run check — "does the cache need updating?" answered deterministically.
  • Self-healing audit that re-validates selectors and flags stale or ambiguous ones.
  • Page Object export and a per-application knowledge base.

03 Authoring

  • Code-first Playwright on the generated Page Object.
  • No-code recorder — record a flow, normalise it into a journey.
  • Plain-English authoring — a described plan rendered to a deterministic spec.
  • Goal-directed, cache-first exploration that drives the browser only for the gaps.
  • Reported as two clear outcomes: goal pass or fail, and whether the cache needs updating.

04 The pull-request gate

  • Traffic-light verdict on every pull request — green, amber or red.
  • Auto-merge on green, review on amber, bug on red.
  • Requirement-justified cache updates — a selector change must fit the requirement.
  • Runs as a GitHub Action, deterministically, with no API key.
  • Configurable base URL — build-and-serve, or a per-branch preview.

05 Quality & traceability

  • Test-trust grading that scores generated tests out of 100.
  • Hallucinated-selector detection against the live application.
  • Requirement traceability back to Jira acceptance criteria.
  • LLM-readable bug reports to Jira or GitHub — with reproduction, de-duplication and link-to-story.
  • Accessibility findings surfaced as a by-product of exploration.

06 Delivery model

  • Available as both a skill and an autonomous sub-agent.
  • Self-hosted — your data stays in your infrastructure.
  • LLM-agnostic — portable across any model or coding agent.
  • Open standards throughout — Playwright, GitHub, Jira.
  • Licence-free, with cost that doesn't scale per seat.
